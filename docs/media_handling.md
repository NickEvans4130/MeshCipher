# Media Handling

MeshCipher supports encrypted image, video, and voice message sharing across all transport layers.

## Encryption

### Algorithm

- **Cipher**: AES-256-GCM (AES/GCM/NoPadding)
- **Key size**: 256 bits (generated per message via `KeyGenerator`)
- **IV**: 96 bits (12 bytes, randomly generated by `Cipher.init()`)
- **Authentication tag**: 128 bits (GCM default)

### MediaEncryptor

```kotlin
// Encryption
fun encrypt(plainBytes: ByteArray): EncryptedMedia {
    val keyGen = KeyGenerator.getInstance("AES")
    keyGen.init(256)
    val secretKey = keyGen.generateKey()

    val cipher = Cipher.getInstance("AES/GCM/NoPadding")
    cipher.init(Cipher.ENCRYPT_MODE, secretKey)

    val iv = cipher.iv
    val encrypted = cipher.doFinal(plainBytes)

    return EncryptedMedia(
        encryptedBytes = encrypted,
        key = Base64.getEncoder().encodeToString(secretKey.encoded),
        iv = Base64.getEncoder().encodeToString(iv)
    )
}

// Decryption
fun decrypt(encryptedBytes: ByteArray, key: String, iv: String): ByteArray {
    val keyBytes = Base64.getDecoder().decode(key)
    val ivBytes = Base64.getDecoder().decode(iv)

    val secretKey = SecretKeySpec(keyBytes, "AES")
    val gcmSpec = GCMParameterSpec(128, ivBytes)

    val cipher = Cipher.getInstance("AES/GCM/NoPadding")
    cipher.init(Cipher.DECRYPT_MODE, secretKey, gcmSpec)

    return cipher.doFinal(encryptedBytes)
}
```

Uses `java.util.Base64` (not `android.util.Base64`) to keep the class unit-testable without Android framework mocks.

## MediaMessageEnvelope

The encrypted media and its metadata are packaged into a single JSON envelope:

```kotlin
data class MediaMessageEnvelope(
    val mediaId: String,          // UUID
    val mediaType: String,        // "IMAGE", "VIDEO", "VOICE"
    val fileName: String,
    val mimeType: String,         // e.g., "image/jpeg", "video/mp4"
    val fileSize: Long,           // Original file size in bytes
    val width: Int?,              // Image/video dimensions
    val height: Int?,
    val durationMs: Long?,        // Audio/video duration
    val encryptionKey: String,    // Base64-encoded AES-256 key
    val encryptionIv: String,     // Base64-encoded 96-bit IV
    val encryptedData: String     // Base64-encoded AES-GCM ciphertext
)
```

## EXIF Metadata Stripping

Before encryption, all images are processed by `MediaProcessor` which:

1. Reads EXIF orientation and applies rotation correction to the bitmap
2. Scales the image to a maximum dimension of 2048px at 90% JPEG quality
3. Strips all identifying EXIF tags from the output file:
   - GPS coordinates, altitude, speed, destination
   - Timestamps and timezone offsets
   - Device make, model, software, serial numbers
   - Author, copyright, user comments
   - Thumbnail data (which can contain its own EXIF)

This prevents metadata-based identification of the sender, their device, or their location.

## Sending Flow

```
1. User selects media (image/video) or records voice message
2. Read raw bytes from content URI or file
2a. (Images only) MediaProcessor strips EXIF metadata and corrects orientation
3. MediaEncryptor.encrypt(rawBytes) -> EncryptedMedia
4. Create MediaMessageEnvelope with:
   - Generated mediaId (UUID)
   - Media metadata (type, dimensions, duration)
   - AES key, IV, encrypted data
5. Serialize envelope to JSON
6. SignalProtocolManager.encryptMessage(envelopeJson, recipientAddress)
   -> Double-encrypted: AES-GCM inner + Signal Protocol outer
7. TransportManager.sendWithFallback(recipientId, signalEncryptedBytes, contentType=1)
```

## Receiving Flow

```
1. Transport layer receives payload with contentType=1
2. SignalProtocolManager.decryptMessage(payload, senderAddress)
   -> Returns MediaMessageEnvelope JSON string
3. Parse JSON to MediaMessageEnvelope
4. MediaEncryptor.decrypt(
     Base64.decode(envelope.encryptedData),
     envelope.encryptionKey,
     envelope.encryptionIv
   ) -> raw media bytes
5. MediaFileManager.saveMedia(
     context, envelope.mediaId, decryptedBytes, envelope.mediaType
   ) -> local file path (e.g., /data/.../media/image/a1b2c3d4...)
6. Create MediaAttachment domain model
7. Save Message entity with mediaId, mediaType, mediaMetadataJson
8. UI renders media inline in ChatScreen
```

## File Storage (`MediaFileManager`)

### At-Rest Encryption

All media files are encrypted at rest using per-file AES-256-GCM. Plaintext never touches the filesystem.

**Encryption flow (on save):**
1. Generate a random 256-bit AES key and let GCM generate a 96-bit IV
2. Encrypt the plaintext bytes with `AES/GCM/NoPadding`
3. Store the per-file key and IV in EncryptedSharedPreferences (`media_encryption_keys`)
4. Write only the ciphertext to disk

**Decryption flow (on display):**
1. Read the per-file key and IV from EncryptedSharedPreferences
2. Read the ciphertext from disk
3. Decrypt with `AES/GCM/NoPadding` using the stored key and IV
4. Return plaintext bytes (images) or write to a temporary cache file (video/voice)

### Directory Structure

```
context.filesDir/
  media_encrypted/
    image/   # AES-256-GCM encrypted files keyed by mediaId
    video/   # AES-256-GCM encrypted files keyed by mediaId
    voice/   # AES-256-GCM encrypted files keyed by mediaId
```

### Operations

```kotlin
fun saveMedia(context: Context, mediaId: String, bytes: ByteArray, mediaType: MediaType): String
fun decryptMedia(mediaId: String, mediaType: MediaType): ByteArray?
fun decryptMediaToTempFile(mediaId: String, mediaType: MediaType): File?
fun getMediaFile(context: Context, mediaId: String, mediaType: MediaType): File?
fun deleteMedia(context: Context, mediaId: String, mediaType: MediaType)
fun cleanupAllMedia(context: Context)
```

`deleteMedia()` removes both the encrypted file on disk and the corresponding key/IV entries from EncryptedSharedPreferences. `cleanupAllMedia()` also removes any legacy unencrypted files from the old `media/` directory.

## Transport-Specific Behavior

### Internet (Direct / Tor Relay)

- Entire `MediaMessageEnvelope` JSON is Signal-encrypted, then Base64-encoded
- Sent as `RelayMessageRequest` with `contentType = 1`
- Relay server stores the opaque blob; it cannot decrypt either layer
- Size limited by relay server's request body limit

### WiFi Direct

- Signal-encrypted envelope bytes are sent via **chunked file transfer**
- `WifiDirectTransport.sendFile()` splits into 64KB chunks
- Receiver reassembles chunks, parses as `MediaMessageEnvelope`
- Chunking allows progress tracking and handles large files

### Bluetooth Mesh

- Media is included in the mesh message payload
- Practical limit: BLE MTU constraints make large media impractical
- The transport manager will prefer WiFi Direct or internet for media when available

### P2P Tor

- Signal-encrypted envelope bytes Base64-encoded into `P2PMessage.payload`
- Sent as `P2PMessage` with `type = MEDIA`
- Single TCP write through SOCKS5 to recipient's .onion address
- No chunking (TCP handles fragmentation)

## MediaAttachment (Domain Model)

```kotlin
data class MediaAttachment(
    val mediaId: String,
    val mediaType: MediaType,    // IMAGE, VIDEO, VOICE
    val fileName: String,
    val fileSize: Long,
    val mimeType: String,
    val localPath: String?,      // Path to encrypted file on disk (decrypted on demand)
    val durationMs: Long?,
    val width: Int?,
    val height: Int?,
    val encryptionKey: String,   // Retained for potential re-encryption
    val encryptionIv: String
)
```

Stored in `MessageEntity.mediaMetadataJson` as serialized JSON. The `localPath` points to the encrypted file under `context.filesDir/media_encrypted/`. The file is decrypted on demand for display.

## Encryption Layering

Media messages have three layers of encryption:

```
Raw media bytes (EXIF stripped for images)
  -> AES-256-GCM at-rest encryption (per-file random key, stored in EncryptedSharedPreferences)
     -> Stored on disk as ciphertext

  -> AES-256-GCM transport encryption (per-message random key)
     -> MediaMessageEnvelope JSON (contains transport key + IV + ciphertext)
        -> Signal Protocol (Double Ratchet session encryption)
           -> Transport layer delivery
```

- **At rest**: Each file is encrypted with its own AES-256-GCM key. The key is stored in EncryptedSharedPreferences, which is itself backed by the Android Keystore. Plaintext never exists on the filesystem.
- **In transit**: The transport encryption key is embedded in the `MediaMessageEnvelope` which is itself encrypted by the Signal Protocol before any transport sees it. The transport key never travels in the clear.
- **EXIF stripping**: All identifying metadata (GPS, timestamps, device info) is removed from images before either encryption layer is applied.
